CWE 란?

: Common Weakness Enumeration

CWE는 Common Weakness Enumeration의 약자로, 소프트웨어 및 하드웨어 취약점 유형 목록입니다.
나열된 단어 자체만 본다면 ‘일반적인 결함의 나열’ 이라고 볼 수 있습니다.
그리고 공통 언어, 보안 도구의 측정 막대, 취약점 식별, 완화 및 예방 노력의 기준선 역할을 합니다.


CWE를 이용하면 정적 분석의 한 부분인 실행시간 오류 ( Runtime Error )를 검출할 수 있습니다.
즉, 프로그램을 실행시키지 않고 분석하면서 오동작을 일으킬 수 있는 코드를 검출 할 수 있게 됩니다.

참고자료 : https://cwe.mitre.org/
https://moasoftware.co.kr/sw-testing/cwe-common-weakness-enumeration-%EB%9E%80/

CWE-119

Improper Restriction of Operations within the Bounds of a Memory Buffer
: 메모리 버퍼 범위 내에서 작업에 대한 부적절한 제한

CWE-119는 버퍼 오버플로우(buffer overflow) 메모리 손상 취약점이다.

메모리 버퍼의 한계 내에서 이뤄지는 운영에 제대로 된 제한을 걸지 못해서 생기는 오류라고 한다.
이 오류를 활성화시킬 경우 메모리를 읽거나 쓰는 여러 행위들이 데이터 구조나 내부 프로그램 데이터와 연결된 메모리 영역에서 의도치 않게 발생하게 된다. 공격자가 이를 활용해 함수 포인터를 악성 코드로 리 다이렉션 하여 악성 코드를 실행하거나, 제어 흐름을 변경하고, 민감한 데이터를 열람할 수 있게 된다.

https://cwe.mitre.org/data/definitions/119.html
참고자료 : http://www.wins21.co.kr/mobile/blog/blog_view.html?t=31&num=1458

CWE-120

Buffer Copy without Checking Size of Input
: 'Classic Buffer Overflow', 입력 크기를 확인하지 않고 버퍼 복사

CWE-120은 프로그램이 보유할 수 있는 것보다 더 많은 데이터를 버퍼에 넣으려고 하거나 프로그램이 버퍼 경계 외부의 메모리 영역에 데이터를 넣으려고 할 때 존재합니다.

가장 단순한 유형의 오류이자 버퍼 오버플로의 가장 일반적인 원인은 프로그램이 복사되는 양을 제한하지 않고 버퍼를 복사하는 "고전적인" 경우입니다. 다른 변종도 존재하지만 고전적인 오버플로의 존재는 프로그래머가 가장 기본적인 보안 보호조차 고려하지 않는다는 것을 강력하게 시사합니다.

참고자료 : https://cwe.mitre.org/data/definitions/120.html

CWE-469

Use of Pointer Subtraction to Determine Size
: 크기 결정을 위한 포인터 감산 사용

CWE-469는 취약한 프로그램의 권한으로 임의의 코드가 실행될 가능성이 있습니다.

응용 프로그램은 크기를 결정하기 위해 다른 포인터에서 하나의 포인터를 감산합니다. 그러나 포인터가 동일한 메모리에 존재하지 않는 경우 해당 작동이 올바르지 않을 수 있습니다.

참고자료 : https://cwe.mitre.org/data/definitions/469.html

CWE-476

NULL Pointer Dereference
: null 포인터 역참조

CWE-476은 응용 프로그램이 유효할 것으로 예상하지만 NULL인 포인터를 역참조할 때 발생하며 일반적으로 충돌 또는 종료를 유발합니다.

이 문제는 경쟁 조건 및 간단한 프로그래밍 누락을 비롯한 여러 결함을 통해 발생할 수 있습니다. 테스트 단계에서 감지를 피할 가능성이 가장 높기 때문에 드물게 발생하는 오류 조건으로 인해 자주 발생합니다.
(여기서 취약점은 보안 관련 취약점을 더 쉽게 도입하거나 탐지하기 어렵게 만들 수 있는 품질 문제입니다.)

참고자료 : https://cwe.mitre.org/data/definitions/476.html
